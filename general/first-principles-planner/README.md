# First Principles Planner

> **第一性原理在产品设计中的应用**

通过深度对话，将产品想法拆解到基本真理，质疑所有功能假设，从核心价值重建最小方案。

---

## 🎯 核心价值

在"我有一个想法"和"这是一个可执行的最小方案"之间架起桥梁，确保你构建的是**真正需要**的功能，而不是**你认为你想要**的功能。

**典型效果**：
- 3-6 个月的项目 → 2-4 周 V1
- 10+ 功能 → 3-5 个核心功能
- 功能蔓延 → 清晰的 V1/V2/V3 划分

---

## 🧠 设计哲学

### 第一性原理 vs 类比思维

**❌ 类比式产品设计**：
```
"做一个像 Notion 的笔记应用"
→ 列出 Notion 的所有功能
→ 模仿现有方案
→ 优化已知方案
→ 结果：3-6 个月，大部分功能用户不需要
```

**✅ 第一性原理产品设计**：
```
"用户的真正问题是什么？"
→ 问题：需要快速记录和找回信息
→ 基本真理：输入 + 搜索 + 存储
→ 最小方案：即时编辑器 + 全文搜索
→ 结果：2-3 周 V1，快速验证假设
```

### 核心原则

1. **功能是假设，不是真理**
   - 大多数功能都未经用户验证
   - V1 的目的是验证，不是完善

2. **核心价值 > 功能数量**
   - 3 个对的功能 > 10 个错的功能
   - 不追求少，追求准

3. **从基本真理重建**
   - 不是在现有方案上简化
   - 而是从核心价值重新设计

## 🛠️ 如何使用

### 触发方式

```bash
# 直接触发
/first-principles-planner

# 自然语言触发
"规划新产品"
"设计 MVP"
"定义功能需求"
"梳理产品思路"
"我想做一个..."
```

### 工作流程（3 个阶段）

#### Phase 1: 拆解（5-10 分钟）

**目标**：识别核心问题和基本真理

```
1. 倾听原始想法（不加评判）
2. 追问核心问题：
   - "用户的真正痛点是什么？"
   - "这个问题的本质是什么？"
3. 拆解到基本真理：
   - 最基本组成是什么？
   - 哪些是物理约束，哪些只是假设？
```

#### Phase 2: 质疑（10-15 分钟）

**目标**：挑战所有功能假设

对每个功能追问：
- "这是'想要'还是'需要'？"
- "没有它，用户还能完成核心任务吗？"
- "有没有更简单的替代方案？"
- "这是基于行业惯例还是实际需求？"

#### Phase 3: 重建（5-10 分钟）

**目标**：基于核心价值定义最小方案

```
1. 识别核心价值
2. 定义最小功能集（3-5 个）
3. 区分 V1（必需）/ V2（验证后）/ V3（长期）
4. 定义验收标准
```

**总时间：15-30 分钟**

## 📋 实际案例

### 案例 1: 笔记应用

**类比思维**：
```
"做一个像 Notion 的笔记应用"

功能：
- 富文本编辑器
- 标签/分类
- 搜索
- 云同步
- 协作编辑
- 版本历史

时间：3-6 个月
```

**第一性原理**：
```
核心问题："用户为什么需要笔记？"
→ 快速记录 + 找回内容

基本真理：
- 输入（快速）
- 存储（可靠）
- 检索（准确）

最小方案：
V1: 纯文本编辑器 + 全文搜索 + 自动保存（2-3 周）
V2: 添加标签
V3: 添加富文本
V4: 添加云同步

时间节省：85%+
```

### 案例 2: 项目管理工具

**类比思维**：
```
"做一个像 Jira 的项目管理工具"

功能：
- 项目/看板/Sprint 视图
- 工作流自定义
- 权限管理
- 报表统计
- 时间追踪
- 集成（Slack, GitHub）

时间：6-12 个月
```

**第一性原理**：
```
核心问题："团队为什么需要项目管理？"
→ 知道谁在做什么 + 进度如何

基本真理：
- 任务列表（谁做什么）
- 状态（进度如何）
- 可见性（团队能看到）

最小方案：
V1: 任务列表 + 状态标签 + 共享链接（3-4 周）
V2: 添加评论
V3: 添加看板视图
V4: 添加复杂权限

时间节省：90%+
```

## 📄 输出格式

生成的 `IMPLEMENTATION_PLAN.md` 包含：

- **核心问题陈述**
  - 用户的真正痛点
  - 问题本质分析

- **基本真理分析**
  - 哪些是基本约束
  - 哪些是被挑战的假设

- **功能划分**（3-5 个核心功能）
  - V1: 绝对必需（验证假设）
  - V2: 验证后添加
  - V3: 长期规划
  - 每个功能的目的、范围、验收标准

- **第一性原理依据**
  - 为什么这样划分
  - 哪些假设被质疑
  - 为什么排除其他功能

## 🎯 适用场景

### ✅ 使用此技能：

- **新产品设计** - 从零设计产品
- **MVP 定义** - 需要快速验证假设
- **功能蔓延** - 产品功能过多，需要重新聚焦
- **创新挑战** - 现有方案不满意，需要突破
- **资源约束** - 时间/预算有限，需要最大化价值

### ❌ 使用其他技能：

- **系统故障/调试** → 使用 [debug](../debug/)（5 Whys）
- **通用问题解决** → 使用 [first-principles](../first-principles/)
- **标准化功能** → 直接执行最佳实践

## 💡 关键洞察

1. **功能是假设，不是真理**
   - 大多数功能都未经用户验证
   - V1 是验证假设，不是完善产品

2. **核心价值 > 功能数量**
   - 3 个对的功能 > 10 个错的功能
   - 不追求少，追求准

3. **类比思维是默认模式**
   - 人类大脑倾向于模仿而非创新
   - 第一性原理需要刻意练习

4. **物理法则是唯一约束**
   - 其他都是可设计的
   - 挑战"一直如此"的惯例
