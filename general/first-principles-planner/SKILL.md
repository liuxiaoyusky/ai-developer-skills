>

> - **通用创新/成本优化**（技术突破、降低成本）→ 用 first-principles

## 🎯 产品设计的第一性原理

### 类比思维 vs 第一性原理

**❌ 类比式产品设计**：
```
"我要做一个像 Notion 的笔记应用"
→ 列出 Notion 的所有功能
→ 模仿现有的解决方案
→ 优化已知方案
```

**✅ 第一性原理产品设计**：
```
"用户的真正问题是什么？"
→ 核心问题：需要快速记录和找回信息
→ 基本真理：输入 + 搜索 + 存储
→ 最小方案：即时编辑器 + 全文搜索
```

## 🧠 产品设计的三步法

### Step 1: 拆解到基本真理

**识别问题的本质**

```
产品想法: "我要做一个带协作功能的项目管理工具"

类比思维:
"像 Jira 一样，需要任务、看板、sprint、报告、权限管理..."

第一性原理:
"用户的真正问题是什么？"
→ 问题：团队不知道谁在做什么，进度如何
→ 本质：信息透明度 + 进度追踪
→ 基本真理：
  - 任务列表（谁做什么）
  - 状态更新（进度如何）
  - 可见性（谁能看到）
```

**关键问题**：
- 用户的真正痛点是什么？
- 这个问题的本质是什么？
- 哪些是物理/技术约束？哪些只是行业惯例？
- 如果没有任何历史限制，最小方案是什么？

### Step 2: 质疑所有功能假设

**挑战"必须有的功能"**

```
假设功能列表:
❌ 用户登录系统
❌ 权限管理
❌ 通知系统
❌ 协作功能
❌ 报表统计

逐一质疑:
"用户登录真的需要吗？"
→ 基本真理：区分不同用户的数据
→ 替代方案：简单的 URL 分享、无需登录的草稿
→ 结论: V1 不需要，后期按需添加

"权限管理真的需要吗？"
→ 基本真理：控制谁能修改什么
→ 替代方案：所有者编辑，其他人只读
→ 结论: V1 用简单角色即可

"协作功能真的需要吗？"
→ 基本真理：多人需要看到同一信息
→ 替代方案：只读共享 + 评论
→ 结论: 先实现信息共享，协作是 V2
```

**关键问题**：
- 这个功能解决的是"想要"还是"需要"？
- 如果没有这个功能，用户还能完成核心任务吗？
- 有没有更简单的替代方案？
- 这个功能是基于"行业惯例"还是"实际需求"？

### Step 3: 从核心价值重建

**基于基本真理设计最小方案**

```
核心价值: 团队了解进度

基本真理:
- 任务列表
- 状态更新
- 可见性

最小重建:
→ V1: 一个简单的任务列表 + 状态标签（所有人可见）
→ V2: 添加评论功能
→ V3: 添加复杂权限

成本: V1 需要 1-2 周（vs 完整版需要 3-6 个月）
```

**关键问题**：
- 基于核心价值，绝对必需的功能是什么？
- 如何用 20% 的工作量实现 80% 的价值？
- 哪些功能可以后期验证后再添加？

## 🛠️ 应用框架

### 工作流程（3 个阶段）

#### Phase 1: 拆解（5-10 分钟）

**目标**: 识别核心问题和基本真理

```
1. 倾听用户的原始想法（不加评判）
   ↓
2. 追问核心问题:
   "用户的真正痛点是什么？"
   "这个问题的本质是什么？"
   ↓
3. 拆解到基本真理:
   "这个问题的最基本组成是什么？"
   "物理/技术约束有哪些？"
   "哪些是不可改变的，哪些只是假设？"
```

**使用 AskUserQuestion**:
- 用户的真正痛点是什么？
- 如果只能保留一个功能，会是什么？
- 没有这个产品，用户现在怎么做？

#### Phase 2: 质疑（10-15 分钟）

**目标**: 挑战所有功能假设

```
对于每个提到的功能:
1. "这个功能解决的是'想要'还是'需要'？"
2. "如果没有它，用户还能完成核心任务吗？"
3. "有没有更简单的替代方案？"
4. "这个功能是基于行业惯例还是实际需求？"
```

**使用 AskUserQuestion**:
- 这个功能是否真的必需？
- 能否用更简单的方式达到同样目的？
- 如果这个功能要砍掉，用户的损失是什么？

#### Phase 3: 重建（5-10 分钟）

**目标**: 基于核心价值定义最小方案

```
1. 基于 Phase 1-2 的发现，识别核心价值
2. 定义实现核心价值的最小功能集（3-5 个）
3. 区分 V1（必需）、V2（验证后）、V3（长期）
4. 为每个功能定义验收标准
```

**使用 AskUserQuestion**:
- 哪些 3-5 个功能是 V1 必需的？
- 哪些功能可以等用户验证后再添加？
- V1 成功的标准是什么？

#### Phase 4: 输出

**生成 IMPLEMENTATION_PLAN.md**，包含：
- 核心问题陈述
- 基本真理分析
- V1/V2/V3 功能划分
- 验收标准
- 第一性原理依据（为什么这样划分）

## ⚠️ 常见误区

### ❌ 误区 1: 第一性原理 = 简化

```
错误: "砍掉一半功能，让产品更简单"
正确: "基于核心价值重建，只保留必需的"

区别:
- 简化: 在现有方案上删减
- 第一性原理: 从核心价值重建
```

### ❌ 误区 2: 第一性原理 = 不做竞品分析

```
错误: "完全忽略竞争对手，闭门造车"
正确: "理解竞品，但不复制"

区别:
- 竞品分析: 了解用户习惯和市场期望
- 类比思维: 直接复制竞品功能
```

### ❌ 误区 3: 第一性原理 = MVP

```
错误: "第一性原理就是做 MVP"
正确: "第一性原理是一种思维方式，MVP 是输出之一"

区别:
- MVP = 最小可行产品（输出）
- 第一性原理 = 如何到达最小方案的思考方法（过程）
```

### ❌ 误区 4: 所有功能都要砍

```
错误: "功能越少越好"
正确: "保留核心价值必需的功能，不追求少，追求准"

判断标准:
- 这个功能是否直接支持核心价值？
- 没有它，核心价值还能实现吗？
```

## 🎯 何时使用

### ✅ 使用第一性原理规划器：

- **新产品设计**：从零设计产品
- **MVP 定义**：需要快速验证假设
- **功能蔓延**：产品功能过多，需要重新聚焦
- **创新挑战**：现有方案不满意，需要突破
- **资源约束**：时间/预算有限，需要最大化价值

### ❌ 不使用（使用其他方法）：

- **纯技术问题**：使用 debug 技能（5 Whys）
- **通用问题解决**：使用 first-principles 技能
- **标准化功能**：明确的行业最佳实践（如支付流程）
- **维护性任务**：已有明确范围

## 💡 关键洞察

1. **功能是假设，不是真理** - 大多数功能都未经用户验证
2. **核心价值 > 功能数量** - 3 个对的功能 > 10 个错的
3. **V1 是验证，不是完善** - 快速验证核心假设
4. **类比思维是默认模式** - 需要刻意练习第一性原理
5. **物理法则是唯一约束** - 其他都是可设计的
