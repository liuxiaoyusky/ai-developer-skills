# 常见的文档过时模式

本文档列出了在审计 Markdown 文档时常见的过时模式，帮助识别可能过时的文档。

## 1. 缺失文件引用

**特征**：
- 文档中引用了不存在的文件路径
- 例如：`见 src/utils/helper.py`，但该文件已被删除或移动

**检测方法**：
- 正则匹配文件路径模式
- 检查文件系统是否存在

**严重程度**：高

## 2. 缺失函数/方法引用

**特征**：
- 文档中提到的函数、方法在代码库中找不到
- 例如：文档描述了 `calculateTotal()` 函数，但代码中已不存在

**检测方法**：
- 从文档提取函数名
- 在代码库中搜索函数定义

**严重程度**：中

## 3. 缺失类引用

**特征**：
- 文档中提到的类在代码库中找不到
- 例如：文档描述了 `UserManager` 类，但已被重构删除

**检测方法**：
- 从文档提取类名
- 在代码库中搜索类定义

**严重程度**：中

## 4. 过时的 API 端点

**特征**：
- 文档中列出的 API 端点已废弃或更改
- 例如：`GET /api/v1/users` 已升级为 `GET /api/v2/users`

**检测方法**：
- 提取 API 端点
- 与当前路由配置对比

**严重程度**：中

## 5. 过时的配置示例

**特征**：
- 配置文件示例使用了已废弃的选项
- 例如：使用了旧版的 config 格式

**检测方法**：
- 检查配置 schema 是否匹配
- 验证配置选项是否存在

**严重程度**：中

## 6. 过时的安装/构建指令

**特征**：
- 使用了已废弃的包管理器命令
- 例如：使用 `npm install` 而项目已迁移到 `pnpm`

**检测方法**：
- 检查项目实际使用的构建工具
- 对比文档中的指令

**严重程度**：低

## 7. 过时的依赖版本

**特征**：
- 文档中指定的依赖版本已过时
- 例如：要求 Node.js 14，但项目已升级到 Node.js 18+

**检测方法**：
- 对比 package.json 或其他依赖文件
- 检查版本要求

**严重程度**：低

## 处理策略

根据严重程度和问题数量决定操作：

### 删除 (delete)
- 高严重程度问题 > 2 个
- 文档内容已完全失效

### 更新 (update)
- 高严重程度问题 1-2 个
- 中严重程度问题 > 3 个
- 文档部分内容仍有效

### 保留 (keep)
- 问题较少或严重程度低
- 文档仍有参考价值

### 人工审核 (manual_review)
- 无法自动判断
- 需要人工确认
